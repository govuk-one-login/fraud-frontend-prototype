Background

For SSF-UI MVP, user accounts are manually created by the SSF-UI dev team. As user numbers of the UI scale, it is impractical for the team to manually create accounts for all users, so users will need a way of creating accounts themselves.

At time of design, there were no users for the Shared signals UI to research with, so the designs have been created following recommendations from our service designer and product managers. Design patterns have also been replicated from existing services like the GOV.UK One Login admin tool.

Super admin onboarding

In order for GOV.UK One Login to control who has access to the UI, a named super admin user from a government organisation will need to be onboarded manually by the dev team over a video call. Designs for this particular journey were contrained to the capabilities of 'cognito', the back end user platform.

User onboarding

Once a super admin exists, they can then invite other users. Following recommendations from our service designer, PM and BA, the access levels are:

  • share and download - a standard access level for most users
  • download only - a user type identified from previous user research, such as service desk users
  • admin - who can share and download, as well as invite or remove 'share and download' or 'download' users

Manage users screen

This is only accessed only be super admin or admin users. It displays a list of the users from the organisation, with the option to remove the user, or withdraw an invite to a user.

For this first phase of account management, it was agreed between UCD and product manager that should any details of an account need to change, like email address, or 2FA method, then the account should be removed and re-created.

To do

  • Use user research to validate the access level options and associated rules (i.e. admin cannot delete admins or super admins)
  • Iterate the designs from insight learnt in research
  • Design future functionality, like the management of users own profile details
